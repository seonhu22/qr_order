## 1. 문서 성격

본 문서는 **현재 저장소 상태 설명 문서가 아닌, 앞으로 적용할 목표 규칙 문서**로 사용한다.

- 현재 구조와 목표 구조는 다를 수 있음
- 목표 기술 스택은 **React 19 / TypeScript / Java 백엔드 연동**
- 프론트엔드는 **멀티앱 구조**를 전제로 설계
- 백엔드와 API 계약이 아직 확정되지 않은 항목은 강제 규칙이 아닌 **권장 사항** 또는 **추후 협의 대상**으로 표기

## 2. 기본 지침

- 답변 및 문서화 언어는 한글 기준
- CLAUDE.md에 정의되지 않은 새로운 디자인 패턴 또는 컴포넌트 추가 시, 사용자 확인 이후 반영
- 승인 후 CLAUDE.md의 UI 디자인 시스템 섹션 업데이트를 선행하고, 이후 코드 작업 진행

## 3. 앱 구조 원칙

프론트엔드 구조는 apps 기반 멀티앱 구성을 목표로 한다.

- admin: 관리자 웹
- client: 사업자 웹/모바일
- consumer: 소비자 모바일 중심

### 3.1 최소 구조 원칙

- 앱별 페이지 코드와 기능 코드는 각 app 하위에서 관리
- 공통 타입은 src 외부의 공용 타입 영역에서 관리 가능
- 상세 폴더 구조는 전환 단계 특성을 고려하여 과도하게 고정하지 않음
- 최소 기준으로 **페이지 단위 코드**와 **기능 단위 코드**는 분리

## 4. TypeScript 기준

- 신규 프론트엔드 코드는 TypeScript 기준 적용
- API 응답에는 타입 선언 포함
- any 사용은 지양
- 불가피한 any 사용 시, 사용 이유를 주석 또는 리뷰 맥락에서 명시
- 컴포넌트 Props는 명시적 타입 선언 적용
- 서버 응답 DTO와 화면 표시용 가공 데이터의 혼용은 지양

## 5. React 19 기준 원칙

- 불필요한 useEffect 사용은 지양
- 파생 가능한 값은 별도 state로 중복 저장하지 않음
- 서버 조회 데이터와 전역 UI 상태는 분리 관리
- 단순 조회 화면에는 **TanStack Query (@tanstack/react-query)** 시범 적용 가능
- 폼 상태 관리 방식은 현재 미확정 상태이며, 추후 별도 규칙으로 정리 예정

## 6. API 통신 규칙

### 6.1 공통 응답 구조

**저장/수정/삭제** API는 @Transactional이 적용되며, ResponseEntity<CommonResponse>를 반환하는 것을 기준으로 한다.  
**단순 조회** API는 데이터를 직접 반환하는 구조를 기준으로 한다.

[공통 응답 JSON 예시]

```
{ "success": true, "message": "성공 메시지" }
{ "success": false, "message": "에러 메시지", "error": "상세 오류" }
```

### 6.2 응답 처리 원칙

- 저장/수정/삭제 응답 (CommonResponse)
  - HTTP 200 + success: true -> 성공 알림 표시
  - HTTP 200 + success: false -> 에러 모달로 message 표시
  - HTTP 200 외 응답 -> 에러 모달로 message 표시, 메시지 없을 경우 기본 문구 사용
  - HTTP 401 / redirect -> 세션 만료 알림 후 / 이동
  - 네트워크 오류 (TypeError) -> 서버 연결 불가 알림 후 새로고침

[응답 처리 로직 예시]

```
fetch(url, options)
  .then((res) => {
    if (res.status === 401 || res.redirected) {
      showAlert("세션이 만료되었습니다. 다시 로그인해주세요.", () => {
        window.location.href = "/";
      });
      return null;
    }
    return res.json().then((data) => ({ ok: res.ok, data }));
  })
  .then((result) => {
    if (!result) return;
    if (result.ok && result.data.success) {
      showAlert(result.data.message || "성공");
    } else {
      showError(result.data.message || "오류가 발생했습니다.");
    }
  })
  .catch((err) => {
    if (err instanceof TypeError) {
      showAlert("서버에 연결할 수 없습니다. 페이지를 새로고침해주세요.", () =>
        window.location.reload(),
      );
    } else {
      showError("오류가 발생했습니다.");
    }
  });
```

### 6.3 요청 데이터 전송

- 저장/수정/삭제 요청은 Content-Type: application/json 바디 전송을 기본으로 설정
- 전송 필드 범위는 백엔드와의 API 계약을 우선 기준으로 삼음
- 생성: 생성 목적에 필요한 명시적 payload 구성
- 수정: 수정 가능한 필드 또는 서버 계약에 따른 명시적 payload 구성
- 삭제: 기본적으로 식별자 중심 전송 우선 검토

### 6.4 GET 파라미터

- GET 조회 파라미터는 쿼리스트링 방식 사용
- encodeURIComponent() 인코딩 적용
- 전체 선택값은 가능하면 ALL 같은 명시적 값 사용 우선 검토
- 빈 문자열과 미전송의 의미 차이는 API 명세 단계에서 백엔드와 협의 필요

## 7. 알림 UI 규칙

- window.alert(), window.confirm(), window.prompt()는 사용하지 않음. 반드시 커스텀 모달 컴포넌트(AlertModal, ConfirmModal, ErrorModal) 사용.

| 상황                 | 사용 모달    |
| :------------------- | :----------- |
| 성공/안내 메시지     | AlertModal   |
| 사용자 확인 필요     | ConfirmModal |
| 서버/클라이언트 오류 | ErrorModal   |

### 7.1 모달 위에서 알림 띄우기

| 상황           | 처리 방식                                  |
| :------------- | :----------------------------------------- |
| 필수값 미입력  | 현재 모달 유지 -> 필드 오류 또는 안내 표시 |
| 서버 저장 성공 | 모달 닫기 -> AlertModal -> 목록 갱신       |
| 서버 응답 오류 | 현재 모달 유지 -> 오류 표시                |
| 세션 만료      | 모달 닫기 -> AlertModal -> / 이동          |

## 8. 메뉴 접근 로그

| 시점      | 엔드포인트                               | 메서드 | 비고                  |
| :-------- | :--------------------------------------- | :----- | :-------------------- |
| 메뉴 진입 | /log/log/menu_open_access_log?menuCd=xxx | POST   | 페이지 마운트 시 실행 |
| 메뉴 이탈 | /log/log/menu_close_access_log           | POST   | 이동 직전 실행        |

- 오류 발생 시 사용자 알림 없이 무시

## 9. 미저장 변경 데이터 보호

인라인 편집 그리드에서 조회 또는 초기화 시점에 미저장 데이터가 존재할 경우 ConfirmModal을 통한 확인 절차 적용.

- 신규 추가 행 또는 수정된 행이 1건 이상인 경우 미저장 데이터로 판단
- 사용자 확인 이후 변경 데이터 폐기 및 조회 진행

## 10. 모달 타이틀 동적 주입

- 신규 모달 타이틀: 현재 메뉴명 + ' 신규 등록'
- 수정 모달 타이틀: 현재 메뉴명 + ' 수정'

### 10.1 메뉴명 관리 원칙

- admin 메뉴명은 JSON 중심 메뉴 메타 관리를 기준으로 설계
- 추후 Client는 서버 상태 데이터 통신을 통해 제작될 예정
- 브라우저 전역 변수(window, globalThis)에 메뉴명을 직접 저장하는 방식은 지양

## 11. 상태 관리 원칙

- 전역 UI 상태와 서버 조회 상태의 혼용은 지양
- 메뉴 상태 및 공용 상태 관리를 위해 Zustand 사용 검토
- 상태 관리 도구 도입은 실제 재사용 범위와 앱 간 영향 범위를 기준으로 확정

## 12. 공용 컴포넌트 원칙

### 12.1 Input 공용화 원칙

- 공통 Input 베이스 컴포넌트 구성 가능
- Base Input에는 과도한 역할을 집중시키지 않음

### 12.2 Base Input에서 지양할 사항

- Props가 과도하게 많아지는 구조
- 로그인/검색/날짜/파일 로직이 한 파일에 모두 들어가는 구조
- variant 분기가 지속적으로 증가하는 구조

## 13. 공용 훅 규칙

- 공용 훅은 UI 렌더링 책임을 직접 가지지 않음
- 특정 화면 전용 훅은 해당 기능 내부에 배치

## 14. 접근 정책 및 라우팅 규칙

- 앱별 라우팅은 분리 관리
- admin 권한 사용자의 client 영역 접근 허용
- client 사용자의 admin 영역 접근 차단

## 15. 디바이스 대응 원칙

- admin 앱: 데스크톱 우선
- client 앱: 반응형 지원
- consumer 앱: 모바일 우선 (QR 진입 사용성 최적화)

## 16. ID 규칙

- 모든 레코드 ID(sysId)는 ULID 형식 문자열로 취급
- Number() 변환 금지, 문자열 그대로 사용

## 17. 브랜치 전략

- main: 안정 브랜치 (직접 수정 금지)
- dev: 개발 통합 브랜치 (직접 push 허용)
- frontend: 프론트엔드 작업 통합 브랜치
- feature/기능명: 기능 개발 브랜치

## 18. 품질 규칙

- lint, typecheck, build 통과 상태를 목표로 함
- 디버깅용 코드 및 사용하지 않는 코드는 정리 후 반영
